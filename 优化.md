## 寄存器优化：

```c
int func3() {
    printf("glo_i = %d\n",i);
    int tt0,tt1,t2,tt3,tt4,v=1906;
    while (i < 10000) {
        int v = a * 4 * 32 * a / a / 32;
        b = func0(b,v);
        tt0 = a*4 + b + c ;
        tt1 = a*4 + b + c + d;
        t2 = a*4 + b + c + d + e;
        tt3 = a*4 + b + c + d + e;
        tt4 = a*4 + b + c + d + e;
        if (func2(i,1000)) {
            printf("sum = %d\n", tt0 + tt1 + t2 + tt3 + tt4);
        }
        func1(i);
    }
    return tt0 + tt1 + t2 + tt3 + tt4;
}
```



1. 总体算法框架
该编译器采用了标准的 图着色寄存器分配算法 (Graph Coloring Register Allocation)，包含构建冲突图、简化、合并、冻结、溢出选择和代码重写等标准阶段。它通过 process() 方法中的循环不断迭代，直到所有变量都被分配了寄存器或被正确溢出处理。
2. 针对 func3 中核心变量 (tt0~tt4) 的分析
在 func3 中，变量 tt0, tt1, t2, tt3, tt4 在 while 循环内被定义，并且其活跃区间（Live Range）跨越了对 func2(i, 1000) 的函数调用。
3. 特定的寄存器约束策略
这是该编译器处理此测试程序的关键特征：
不使用/不支持 Callee-Saved 寄存器跨调用保持：代码中缺乏在函数序言/尾声保存和恢复被调用者保存寄存器（如 $s0-$s7）的逻辑。
函数调用即“全杀”：为了保证正确性，build 阶段构建冲突图时，call 指令被视为定义（破坏）了所有可分配的物理寄存器。
4. 具体分配流程 (func3 场景演绎)
冲突构建 (Build)：
由于 tt0~tt4 在 func2 调用时是活跃的（Live）。
func2 调用指令与所有可分配寄存器产生写冲突。
结果：tt0~tt4 在冲突图（Adjacency Graph）中与每一个物理寄存器都存在连边。
着色尝试 (AssignColors)：
算法尝试为 tt0 分配颜色。
检查邻接点时，发现它与所有物理寄存器冲突。
okColors 集合为空。
结果：tt0~tt4 直接被判定为无法着色，加入 spilledNodes 集合。此处实际上跳过了基于代价（selectSpill）的溢出选择，因为是硬性无法分配。
溢出代码重写 (RewriteProgram)：
编译器决定将 tt0~tt4 溢出到栈上。
栈空间：通过 func.addAllocaSize(4) 为它们在栈帧中分配槽位。
指令插入：
在计算出 tt0 的指令后，插入 sw vReg, offset($sp)。
在打印语句使用 tt0 前，插入 lw vReg, offset($sp)。
引入极短生命周期变量 (vReg)：
为了执行上述存取，代码引入了新的虚拟寄存器 vReg。
这些 vReg 的生命周期非常短（仅限于加载/存储和相邻的一条计算指令），不再跨越函数调用。
再迭代 (Re-process)：
算法重新运行。
由于新生成的 vReg 不跨越函数调用，它们只与少量的寄存器冲突。
编译器轻松地将 Caller-Saved 寄存器（如 $t0 等）分配给这些 vReg。
总结
对于 func3，您的编译器的策略是：放弃将跨越函数调用的变量（tt0~tt4）保留在寄存器中，强制它们溢出到内存。
它通过在每次定义后写栈、每次使用前读栈的方式，利用生命周期极短的临时寄存器来完成计算，从而规避了维护 Callee-Saved 寄存器的复杂性。虽然牺牲了部分性能（频繁访存），但保证了代码生成的正确性。

## mem2reg

这个过程主要分为三个阶段：局部优化、特殊情况处理、以及通用的 SSA 构建（插入 Phi 节点与变量重命名）。
以下是针对您代码逻辑的详细步骤拆解：
1. 局部优化：基本块内的清理 (sweepBlock)
在正式构建 SSA 之前，代码先对每个基本块进行了一次“扫除”，目的是做 基本块局部的死代码消除和 Store-to-Load 转发。
Store-to-Load 转发：
在一个基本块内，如果先 store v1, ptr，紧接着 load v2, ptr，代码会直接用 v1 替换 v2 的所有使用，并删除该 load 指令。
实现细节：通过 HashMap<Alloca, Store> alloca2store 记录当前块内最近一次对某 alloca 的 store。
死存消除 (Dead Store Elimination)：
如果在同一个块内，对同一个地址进行了多次 store，且中间没有 load，那么只有最后一次 store 是有意义的，前面的可以删除。
实现细节：代码最后有一个逆序遍历（for (int i = instructions1.size()-1 ...)），如果发现连续的 store，则删除被覆盖的前者。
2. 信息收集与特殊情况处理 (Fast Paths)
对于每个候选的 alloca，代码会先调用 analyzeAllocaInfo 收集 定义点（Store所在块） 和 使用点（Load所在块）。
为了提升编译速度，您实现了两种“快路径（Fast Path）”来跳过复杂的 Phi 节点计算：
A. 只有一次 Store (dealOnlyStore)
场景：全局只有一个地方对该变量赋值。
处理：
如果那个唯一的 store 支配（Dominate） 了 load 所在的块，直接用 store 的值替换所有的 load。
这种情况下不需要 Phi 节点，因为值只有一种来源。
B. 局限在单个基本块内 (dealOneBlock)
场景：变量的定义和使用都完全局限在一个基本块内（例如局部临时计数器）。
处理：
利用 sweepBlock 的成果，此时基本块内应该已经只剩下有效的计算逻辑。
直接删除 alloca 及其相关的指令，因为局部转发已经完成。
3. 通用 SSA 构建：插入 Phi 节点
如果上述特殊情况都不满足，说明变量在控制流图（CFG）中存在复杂的跨块依赖，需要插入 Phi 节点。
计算支配边界 (calIDF)：
使用 迭代支配边界 (Iterated Dominance Frontier, IDF) 算法。
根据所有定义该变量的块（definingBlocks），找出它们的支配边界闭包。这些边界就是控制流汇聚的地方，也是需要插入 Phi 节点的地方。
活跃性检查 (isPhiAlive)：
这是一个剪枝优化。如果计算出的边界块中根本没有在这个 alloca 上发生 load，或者在此之前已经被重新 store 覆盖了，就不需要插入 Phi。
插入 Phi：
在确定的边界块头部插入空的 Phi 指令，并记录 phi2Alloca 映射，供后续重命名使用。
4. 变量重命名 (renamePhiNode)
这是 Mem2Reg 的核心步骤，完成了从内存变量到 SSA 版本的转换。
算法逻辑：
使用一个 栈 (Stack) 模拟对控制流图（CFG）的深度优先遍历 (DFS)。
维护一个 variableVersion 映射：Current State: Map<Alloca, Value>，表示当前路径上每个 alloca 对应的最新 SSA 值。
处理流程：
处理 Phi 定义：进入块时，如果块头有新插入的 Phi，将其作为该 alloca 的最新版本存入 variableVersion。
指令重写：
遇到 Load：查询 variableVersion，用当前的最新值替换 Load，删除 Load。
遇到 Store：更新 variableVersion，将 Store 的值记为最新版本，删除 Store。
填充后继块的 Phi：
查看当前块的所有后继块（Successors）。
如果有后继块包含属于该 alloca 的 Phi 节点，将当前块产生的最新值（Incoming Value）和当前块标签（Incoming Block）填入 Phi 指令。
递归：携带当前的 variableVersion 副本继续遍历后继块。
总结您的实现特点
激进的清理策略：sweepBlock 的存在使得在进入复杂的 SSA 逻辑前，大量的冗余 Load/Store 就已经被消除了，这既减少了图的规模，也让后续分析更简单。
标准的 IDF + Renaming：采用了经典的 Cytron 等人提出的 SSA 构建算法的核心思想（基于支配边界插入 Phi，基于 DomTree/CFG 遍历进行重命名）。
栈模拟递归：在 renamePhiNode 中手动维护栈结构来保存 variableVersion 的状态快照，避免了深层递归可能导致的栈溢出问题，同时清晰地表达了作用域的层级关系。
Java 数据结构的使用：通过 HashSet 和 HashMap 有效管理 def/use 链，但有一点需要注意：rewriteProgram 或 sweepBlock 中对 LinkedList 使用索引访问 (get(i)) 可能会导致 <span>O(N^2)</span> 的复杂度，如果是大型基本块可能会成为性能瓶颈（建议使用迭代器或转为 ArrayList）。
